# 大厂招聘官网可见岗位分类层级调研

调研日期：2026-07-18

## 结论

这些招聘官网没有一套可以直接共用的岗位分类：美团、字节、腾讯在职位筛选器中提供明确的两级职类；阿里和拼多多的“职位类别”只有一级，页面上的业务集团、招聘项目、职位标签属于并列筛选维度，不能当成职位二级。

OfferLens 应采用两套分开的数据：

1. `official_category`：忠实保存官网页面可见的一级、二级名称；官网没有二级时保持为空。
2. `normalized_taxonomy`：把不同公司的官网分类映射为统一的 `role_group → role_family`。官网没有二级时，可以依据岗位标题和 JD 推断，但必须记录 `inferred`、证据和置信度，不能伪装成官网分类。

## 官网页面证据

职位类别使用两级联动筛选。

| 官网一级 | 本次实际展开看到的官网二级 |
| --- | --- |
| 技术类 | 测试、运维、算法、硬件、软件 |
| 产品类 | 硬件产品、产品 |
| 商业分析类 | 商业分析 |
| 零售类 | 物流、采购、供应链、商品运营、门店 |
| 运营类 | 本次未展开，不作推测 |
| 设计类 | 本次未展开，不作推测 |
| 市场营销类 | 本次未展开，不作推测 |
| 职能类 | 本次未展开，不作推测 |
| 金融类 | 本次未展开，不作推测 |
| 销售、客服与支持类 | 本次未展开，不作推测 |
| 综合类 | 本次未展开，不作推测 |

判断：官网有可靠 L1、L2；但“软件”“算法”等二级仍较粗，OfferLens 可在统一分类中继续依据标题/JD细分为后端、前端、客户端、机器学习工程等。

### 字节跳动

页面：[字节跳动社会招聘职位列表](https://jobs.bytedance.com/experienced/position?keywords=&category=&location=&project=&type=&job_hot_flag=&current=1&limit=10&functionCategory=&tag=)

职位类别使用一级职类与二级职能联动筛选。

- 官网一级：研发、运营、产品、职能/支持、设计、市场、游戏策划、销售。
- 展开“研发”后可见二级：算法、后端、前端、客户端、测试、机器学习、大数据、安全、硬件、基础架构、多媒体、计算机视觉、运维、数据挖掘。

判断：这是本次页面证据中最完整、最接近 OfferLens 统一技术分类的两级结构，可作为技术岗位归一化的主参考，但仍需把“计算机视觉”视作技术专业方向，而不是与所有岗位族并列的唯一标准。

### 腾讯

页面：[腾讯校招岗位投递](https://join.qq.com/post.html?query=p_2)

职位类别使用一级职类与二级岗位方向联动筛选。

- 官网一级：青云课题、技术、产品、设计、市场、职能。
- 展开“技术”后可见二级：软件开发类、技术运营类、安全技术类、测试与质量管理类、算法研究类、解决方案与服务类、硬件开发类。

判断：官网 L2 有效但较粗。例如“软件开发类”仍需在统一分类中依岗位标题/JD拆为后端、前端、客户端、游戏开发等。

### 阿里巴巴

页面：[阿里巴巴校园招聘职位列表](https://campus-talent.alibaba.com/campus/position?batchId=100000700001&filterParams=%7B%22aliStar%22%3A%5B%22Y%22%5D%7D)

“职位类别”为单层复选框：技术类、游戏类、运营类、设计类、产品类、物流类、销售类、内容类、营销类、数据类、职能类、风险管理类、其他。

页面另有“业务集团/公司”等并列筛选器，但它描述组织归属，不是所选职位类别的下一级。

判断：页面可确认的职位分类只有 L1。技术类岗位的后端、算法、数据、基础架构等 L2 必须由 OfferLens 根据岗位标题/JD推断，并明确标记为推断结果。

### 拼多多

页面：[拼多多校园招聘](https://careers.pddglobalhr.com/campus/grad)

“职位类别”为单层复选框：设计、职能、运营、语言、市场营销、产品、技术、视觉类、区域业务。

页面另有“招聘项目”和“职位标签”，二者是并列筛选维度，不是职位类别的下一级。

判断：页面可确认的职位分类只有 L1。技术类的具体方向需要从标题/JD推断。

## OfferLens 统一双层模型建议

统一模型不追求复刻任意一家官网，而是让跨公司搜索、简历匹配和面经分析使用同一语义。

### 统一一级 `role_group`

建议保持少而稳定：

| 统一一级 | 可承接的官网一级示例 |
| --- | --- |
| 技术 | 技术类、研发、技术、数据类 |
| 产品 | 产品类、产品 |
| 运营 | 运营类、运营、商品运营 |
| 设计 | 设计类、设计、视觉类 |
| 市场/销售/服务 | 市场、市场营销类、销售、销售类、销售、客服与支持类 |
| 商业/供应链 | 商业分析类、零售类、物流类、区域业务 |
| 职能 | 职能类、职能/支持、风险管理类、语言 |
| 内容/游戏 | 内容类、游戏类、游戏策划 |
| 其他 | 综合类、其他、暂时无法可靠映射的类别 |

### 技术一级下的统一二级 `role_family`

| 统一技术二级 | 直接官网证据 | 需要补充判断的情形 |
| --- | --- | --- |
| 后端开发 | 字节“后端” | 美团“软件”、腾讯“软件开发类”、阿里/PDD“技术”需看标题/JD |
| 前端开发 | 字节“前端” | 其他公司的“软件/技术”需看标题/JD |
| 客户端开发 | 字节“客户端” | 腾讯“软件开发类”等需看标题/JD |
| 算法 | 美团“算法”、字节“算法”、腾讯“算法研究类” | 可再用专业标签表达推荐、搜索、NLP等方向 |
| 机器学习工程 | 字节“机器学习” | “算法/软件/技术”中的模型训练、推理、平台岗位需看 JD |
| 数据工程 | 字节“大数据”“数据挖掘” | 阿里“数据类”及数据平台岗位需看标题/JD |
| 测试与质量 | 美团“测试”、字节“测试”、腾讯“测试与质量管理类” | — |
| SRE/运维 | 美团“运维”、字节“运维”、腾讯“技术运营类” | “技术运营类”可能含非 SRE 岗位，需结合 JD |
| 安全 | 字节“安全”、腾讯“安全技术类” | 合规、风控等非安全研发岗位不能仅凭“安全”二字归入 |
| 硬件 | 美团“硬件”、字节“硬件”、腾讯“硬件开发类” | 可用专业标签补充芯片、嵌入式、服务器等 |
| 基础架构/网络 | 字节“基础架构” | 美团“软件”、腾讯“软件开发类”及其他技术类需看标题/JD |
| 多媒体/视觉 | 字节“多媒体”“计算机视觉” | 统一二级可合并，具体方向放专业标签 |
| 解决方案/技术服务 | 腾讯“解决方案与服务类” | 美团“销售、客服与支持类”中只有 JD 明确技术交付时才归入 |
| 游戏研发 | 阿里“游戏类”、字节“游戏策划”只能作为候选证据 | 必须区分研发、策划、运营，最终看标题/JD |

## 数据边界

推荐最小记录结构：

```text
official_category.level1.name
official_category.level2.name       # 官网页面没有则为 null
official_category.source_url

normalized_taxonomy.role_group
normalized_taxonomy.role_family
normalized_taxonomy.method          # official_mapping | inferred
normalized_taxonomy.evidence        # 官网路径，或命中的 title/JD 片段
normalized_taxonomy.confidence
normalized_taxonomy.version
```

规则：

- 官网 L1/L2 原样保存，不因统一分类而覆盖。
- 官网 L2 存在时，优先通过显式映射得到统一岗位族。
- 官网只有 L1 时，`official_category.level2` 必须为 `null`；标题/JD推断只写入 `normalized_taxonomy`。
- 业务集团、招聘项目、职位标签分别作为组织、项目、标签维度保存，不提升为职位分类层级。
- 页面未展开的二级选项不猜测；后续抓取或页面复核后再补。
