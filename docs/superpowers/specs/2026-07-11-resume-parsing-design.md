# 简历解析功能设计

## 目标

让用户上传 PDF 简历后，获得可核对的姓名、手机号、邮箱和经历分区，并保留完整文本编辑与确认能力。使用 `简历/` 目录中的 5 份真实样本作为回归数据。

## 方案选择

采用本地确定性解析：继续使用已安装的 `pypdf`，将文本提取切换为 `extraction_mode="layout"`，再用固定规则提取基础信息和常见分区。

不采用以下方案：

- OCR：5 份样本都有可用文本层，不需要额外模型、系统命令或部署依赖。
- AI 结构化解析：会增加费用、延迟和个人信息外发，本阶段没有必要。
- 只返回原始文本：现有实现已经具备，无法满足用户核对解析结果的目标。

## 后端

将 PDF 文本提取与结构解析放到独立服务函数，路由只负责文件校验和响应。

解析流程：

1. 校验扩展名、PDF 文件头和 8 MB 大小上限。
2. 最多读取前 12 页。
3. 每页优先使用布局模式提取，兼容性异常时回退普通 `extract_text()`。
4. 统一空白与空行，保留自然段和分区边界。
5. 用正则提取手机号、邮箱。
6. 在首个经历分区前查找 2–4 个中文字符的姓名，排除已知分区标题；无法识别时返回空字符串。
7. 按教育、工作、实习、项目、技能、荣誉、校园、个人总结等标题切分内容。

接口保持 `POST /api/recruitment/resume/parse`，响应扩展为：

```json
{
  "text": "完整可编辑文本",
  "name": "姓名",
  "phone": "手机号",
  "email": "邮箱",
  "page_count": 1,
  "char_count": 2145,
  "sections": [
    { "key": "education", "title": "教育经历", "content": "..." }
  ]
}
```

保留 `text` 字段，现有前端和 AI 报告请求继续兼容。

## 前端

上传成功后继续使用现有简历解析结果卡，增加：

- 姓名、手机号、邮箱、页数、文本字数。
- 已识别分区卡片，显示标题和内容摘要。
- 完整解析文本编辑框，仍是最终送入 AI 报告的数据源。
- 用户修改完整文本后撤销确认状态；结构化摘要标记为“上传时识别结果”，避免暗示会随文本编辑实时重算。

直接粘贴简历文本保持现有流程，不调用 PDF 解析接口，也不强制生成结构化摘要。

## 错误处理

- 非 PDF 或伪装文件：提示“请上传有效的 PDF 简历”。
- 超过 8 MB：提示“PDF 文件不能超过 8MB”。
- 加密、损坏或读取异常：返回可读的解析失败信息。
- 无文本层：明确提示需要文本型 PDF，不静默返回空内容。
- 未识别姓名或分区：返回空字段或空数组，完整文本仍可编辑确认。

## 测试与验收

- 服务层单元测试覆盖布局提取、姓名/联系方式、分区切分和无文本错误。
- 使用 `简历/` 下 5 份 PDF 逐一回归：均能提取非空文本、姓名、手机号和邮箱；至少识别一个经历分区。
- API 测试覆盖文件头、大小和兼容响应。
- 前端契约测试覆盖新响应字段和解析结果展示。
- 运行后端测试、语法检查、前端测试与生产构建。
- 在浏览器上传样本 PDF，验证字段、分区、原文编辑、修改后撤销确认和最终确认流程。

## 不做

- 扫描件 OCR。
- Word 简历解析。
- AI 自动改写或自动补全缺失字段。
- 保存原始简历文件或结构化结果到数据库。
